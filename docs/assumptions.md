# Assumptions Log

Running log of assumptions made across Vision, Concept, feature docs, and ADRs — surfaced explicitly rather than left implicit in conversation. Add an entry whenever a decision depends on something unverified.

| Date | Assumption | Status | Notes / Evidence |
|---|---|---|---|
| 2026-07-23 | Fully manual tracking (no passive/idle detection) will still deliver "no work forgotten" in practice, since nothing enforces that the user remembers to start/switch/interrupt tracking. | open | Explicitly scoped this way for MVP (determinism, privacy, user control) per `docs/vision/vision.md`; revisit after real personal usage. Tracked as risk R3 in `docs/risks.md` (status: accepted). |
| 2026-07-23 | Auto-completing all skipped intermediate interruptions when the user chooses "Return to Original Task" matches real user intent every time. | open | Initially flagged as plausibly contradicting Vision's "never silently dropped" success criterion (reviewer pass, 2026-07-23). Resolved at the design level via a `completion_reason` field (`explicit` vs `auto-completed-on-skip`) so skipped work stays reviewable rather than indistinguishable from finished work — see `docs/concept/concept.md` and `docs/glossary.md`. Whether this actually matches real user intent in practice remains open; tracked as risk R1 in `docs/risks.md` (status: mitigated by design, not yet implemented). |
| 2026-07-23 | Export-time aggregation by exact match on task name/project/client (rather than a first-class Task entity with identity) is sufficient for useful billing reports. | open | Broadened per reviewer pass (2026-07-23): risk isn't just typos, but any rename of a task/project/client over time silently fragmenting historical totals, with no reconciliation path. Task Templates partially mitigate by giving a canonical source for names. Tracked as risk R2 in `docs/risks.md` (status: open). Revisit once export/reporting is actually designed. |
| 2026-07-23 | A desktop-first app with global hotkeys is sufficient for "effortless" capture, even for interruptions that happen away from the desk (phone calls, in-person conversations), until a mobile client exists. | open | Mobile/CLI/browser clients are explicitly deferred; this assumption is only really testable once real usage away from the desk is observed. |
| 2026-07-23 | Existing time-tracking tools (Toggl, Harvest, Clockify, RescueTime) do not adequately solve the interruption-stack/return-to-original-task problem — i.e., this is a genuine gap, not a reinvention. | open | No competitive research done yet; see `docs/concept/concept.md` "How it's different" and `research/`. |
| 2026-07-23 | The author's prior Rust experience is sufficient that the Rust surface area chosen in ADR 0002 (hotkeys, tray, window management, log I/O/compaction, heartbeat timer, OS-level sleep/hibernate detection via raw Windows APIs) is a manageable ramp-up cost, not a project-stalling one. | open | Not stated either way during architecture discussion; treated as an open assumption rather than guessed at. Tracked as risk R6 in `docs/risks.md` (status: accepted). Revisit once real implementation work on the Rust core begins. |
| 2026-07-23 | Billable-vs-non-billable classification does not need to be tracked as an explicit field in Anchor's data model — project/client attribution alone is sufficient, because the exported timeline is transferred into a separate, already-used billing/invoicing process that performs the actual classification. | resolved-by-decision | Not an evidence-based validation — this was a scope question only the author could answer, settled directly rather than through research or usage. See [ADR 0003](decisions/0003-billable-classification-out-of-scope.md), which is the actual resolution mechanism. Resolved a real gap: `docs/vision/vision.md` "Success looks like" previously implied Anchor itself splits billable/non-billable time, but no such field was ever defined on Time Block (`docs/concept/concept.md`). Vision, MVP, and Users docs amended the same day. |
| 2026-07-23 | JSON export should stay a raw, per-Time-Block record list (independently rounded per record) rather than grouped by task the way XLSX is. | invalidated | This was the initial default design call in `docs/product/features/export.md`, flagged as unconfirmed rather than treated as decided. Reviewer pass (2026-07-23) found it would cause XLSX and JSON to diverge numerically (~3x) for identical data in the exact fragmented-interruption scenario this feature exists for. Author corrected it directly: JSON uses the same "sum then round" approach as XLSX when rounding is enabled, staying raw only when rounding is disabled. See `docs/product/features/export.md` "JSON export shape." |

- **open** — not yet checked
- **validated** — confirmed by research, user feedback, or real usage; safe to build on
- **invalidated** — turned out false; note what changed as a result (which docs/ADRs needed revisiting)
- **resolved-by-decision** — not evidence-based; settled by an explicit author or architecture decision (cite the ADR). Distinct from `validated`: nothing was empirically confirmed, a scope call was made. Only revisit if the decision itself is revisited (new ADR), not through further research.
