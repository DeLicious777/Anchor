# Risk Register

Risks identified across product, UX, technical, and operational dimensions. A risk earns a row here when it could plausibly derail the Vision, MVP, or a specific feature/ADR — not every hypothetical worry.

| ID | Risk | Likelihood | Impact | Mitigation / Owner | Status |
|---|---|---|---|---|---|
| R1 | Auto-completing skipped interruptions (Return to Original Task) could mask genuinely unfinished work if indistinguishable from an explicit completion. | med | med | Time Blocks record a `completion_reason` (`explicit` vs `auto-completed-on-skip`), keeping skipped work reviewable/reopenable rather than silently equated with finished work. See `docs/concept/concept.md` "Core concept" and `docs/glossary.md`. | mitigated (by design; not yet implemented) |
| R2 | Export-time aggregation by exact match on name/project/client can silently fragment historical billing totals if a task's name, project, or client is renamed or entered inconsistently over time — no persisted Task identity exists to catch this. | med | med | Partially mitigated going forward by Task Templates (`docs/product/features/task-templates.md`): consistent selection from a template reduces *entry-time* typos/drift for new Time Blocks. Does **not** close the underlying gap — editing an existing template's name/project/client explicitly does not retroactively update Time Blocks already recorded under the old values (by design, per the flat-timeline model), so a template rename mid-life reproduces the same fragmentation this risk describes, just less often. No reconciliation/rename tool exists yet. This exact-match grouping is now concretely implemented (not just described) in `docs/product/features/export.md`'s XLSX row structure — the risk becomes visible directly as under/over-reported task rows in the exported sheet. Revisit if real usage shows fragmentation. | open |
| R3 | Fully manual tracking (no idle/gap detection in MVP) depends entirely on user discipline; nothing prevents forgetting to log a Switch/Interrupt, which directly undermines the Vision's "accurate record... no manual reconciliation" promise. | med-high | high | None in MVP — consciously deferred for determinism/privacy/simplicity (see `docs/vision/vision.md` Non-goals). To be revisited after real personal usage; see `docs/assumptions.md` row 1. | accepted |
| R4 | A `recovered-gap` entry's inferred end time can be wrong if the user never notices/corrects it in the dashboard before exporting, undermining "trustworthy... without hand-editing." Depends on the same user-discipline gap as R3. | med | high | A fixed 60-second heartbeat write while an entry is active bounds the inference error to roughly a minute (not hours), and the dashboard surfaces `recovered-gap` entries distinctly (not folded into `explicit`), inviting correction — but nothing enforces that the user actually reviews/corrects them before exporting. `docs/product/features/export.md` explicitly considered and rejected enforcing review at export time (author's own workflow discipline instead) — this is now a confirmed design choice, not an oversight. | open (staleness bounded by design; review still not enforced by choice, not omission) |
| R5 | A torn write (process killed mid-write to the persisted transition log) could corrupt more than just the latest transition if the storage format isn't append-only/atomic at the record level. | low | high | Mitigated by design: append-only transition log means a torn write can only affect the last, unreplayed record, which is detected (fails to parse) and discarded on next launch — every prior record is untouched. Concretely specified in [ADR 0004](decisions/0004-transition-log-format-and-torn-write-scheme.md): JSON Lines with a checksum framed outside the JSON object (avoiding a self-reference bug an earlier draft of that ADR had), parse+checksum validated on read. | mitigated (by design; not yet implemented) |
| R6 | Tauri's ecosystem is younger/less battle-tested than Electron's, and this is a solo project with a real (not "narrow") Rust surface area — hotkeys, tray, window management, log I/O/compaction, heartbeat timer, and OS-level sleep/hibernate detection (which requires raw Windows API calls, not a Tauri-provided feature). For a single maintainer, ecosystem gaps or unexpected platform-API complexity could stall MVP progress more than a more mature/higher-level stack would. | med | med | None specific yet — accepted as the cost of satisfying the memory/startup priorities and frontend-stack constraint together (see ADR 0002). Revisit ADR 0002 if `tauri-plugin-global-shortcut` lacks conflict detection or sleep/hibernate detection proves substantially harder than expected. | accepted |
| R7 | Compaction (snapshot + log truncation, [ADR 0004](decisions/0004-transition-log-format-and-torn-write-scheme.md)) introduces a second on-disk artifact whose own write can be interrupted — a crash mid-snapshot-write, or between a successful snapshot write and log truncation, could otherwise cause double-replay or data loss if handled naively. | low | high | Mitigated by design: a watermark (sequence number) recorded in each snapshot makes replay correct regardless of when a crash occurs relative to truncation — log truncation becomes a disk-space optimization, not a correctness dependency. Snapshot writes use write-temp-then-atomic-rename. See ADR 0004 Consequences. | mitigated (by design; not yet implemented) |

- **mitigated** — a concrete mitigation is in place
- **accepted** — consciously not mitigating; the cost of mitigation exceeds the risk
- **realized** — it happened; document what actually occurred and link any resulting ADR or assumption update

**Keeping this current:** review this register whenever a new ADR or feature doc is accepted — ask whether it introduces a new risk or changes the likelihood/impact of an existing one.
